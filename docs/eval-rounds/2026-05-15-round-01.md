## How to drive a session

One **set** = one `(target, question)` row × 3 trials × one arm.
Each set needs **two** sessions: arm A first, arm C second (arm-C
also runs the judges, the summary, and the commit).

### Per-session preflight (your shell, *before* launching `claude`)

Export the env vars the harness hooks read:

```bash
export SEER_EVAL_RUN_ID=2026-05-15-round-01
export SEER_EVAL_ARM=A     # or C, depending on which session you're starting
```

For an **arm-A session only**, also disable the `repo-map` skill so the
operator-Claude can't even see it (defense-in-depth on top of the
prompt-level forbid):

```bash
# arm-A only:
mv .claude/skills/repo-map .claude/skills/_repo-map.disabled
```

Then launch claude in the seer-cli repo root and paste the matching
section below.

After the arm-A session, restore the skill before the arm-C session:

```bash
mv .claude/skills/_repo-map.disabled .claude/skills/repo-map
```

## Arm-A procedure

**Before starting**, verify env state and skill state:

```bash
env | grep -E "^SEER_EVAL_(RUN_ID|ARM)="
# expect: SEER_EVAL_RUN_ID=2026-05-15-round-01
#         SEER_EVAL_ARM=A
```

Confirm the available-skills list at session start does **not** include
`repo-map`. If it does, stop and ask the user to relaunch with the skill
disabled (see preflight above).

If this is the first set of the run, also (idempotent):

```bash
uv run --group experiments python -m experiments.scripts_eval.manifest \
    init --run $SEER_EVAL_RUN_ID
```

**For each trial in {1, 2, 3}:**

1. Read the question template for `{QUESTION_ID}` from
   `experiments/scripts_eval/corpus.yaml`. Look up the path for
   `{TARGET}` from the same file.

2. Substitute `{repo_path}` (or `{workspace_root}` for the workspace
   question) in the template, then append:

   ```text

   Constraints (verbatim):
   - You may NOT use the `repo-map` skill. You may NOT invoke
     `python -m seer.repo`, the `seer.repo` Python module, or any
     `scripts/*.sh` paths under `.claude/skills/repo-map/`.
     If you cannot answer without them, say so explicitly and stop.
   - Use only Read, Grep, Glob, and Bash.
   - After answering, append two sections and stop:
     ### tools_used
     - <ToolName>: <count>  (one line per distinct tool)
     ### evidence
     - <one path per line>
   ```

3. Dispatch **one** `Explore` subagent with that full prompt.

4. After the subagent finishes, run capture:

   ```bash
   uv run --group experiments python -m experiments.scripts_eval.capture \
       --run $SEER_EVAL_RUN_ID --repo {TARGET} \
       --question {QUESTION_ID} --trial {N}
   ```

   (For the workspace question, omit `--repo`.)

5. Confirm a new JSON exists at
   `experiments/scripts_eval/results/$SEER_EVAL_RUN_ID/arm-A/{TARGET}-{QUESTION_ID}-t{N}.json`.

**After all 3 trials**, update the tracked evidence file and commit:

```bash
uv run --group experiments python -m experiments.scripts_eval.summarize \
    --run $SEER_EVAL_RUN_ID \
    --out docs/eval-rounds/2026-05-15-round-01.md

git add docs/eval-rounds/2026-05-15-round-01.md
git commit -m "round-01: arm-A captured for {TARGET}/{QUESTION_ID} (3 trials)"
```

Report back: number of cells now under arm-A/, what's pending for arm C
on this set, and the next suggested set per the run-state table.

## Arm-C procedure

**Before starting**, verify env state and the arm-A precondition:

```bash
env | grep -E "^SEER_EVAL_(RUN_ID|ARM)="
# expect: SEER_EVAL_RUN_ID=2026-05-15-round-01
#         SEER_EVAL_ARM=C

ls experiments/scripts_eval/results/$SEER_EVAL_RUN_ID/arm-A/{TARGET}-{QUESTION_ID}-t*.json
# expect: 3 files (t1, t2, t3)
```

If the precondition fails, stop — arm A must complete first.

**For each trial in {1, 2, 3} — tester phase:**

1. Substitute the question template (same as arm A) but with the
   arm-C rider:

   ```text

   Constraints (verbatim):
   - You may use the `repo-map` skill and its scripts at your discretion.
   - After answering, append two sections and stop:
     ### tools_used
     - <ToolName>: <count>
     ### evidence
     - <one path per line>
   ```

2. Dispatch one `Explore` subagent.

3. Capture:

   ```bash
   uv run --group experiments python -m experiments.scripts_eval.capture \
       --run $SEER_EVAL_RUN_ID --repo {TARGET} \
       --question {QUESTION_ID} --trial {N}
   ```

**For each trial in {1, 2, 3} — judge phase:**

1. Prepare the blinded job:

   ```bash
   uv run --group experiments python -m experiments.scripts_eval.judge \
       prepare --run $SEER_EVAL_RUN_ID \
       --pair-key {TARGET}/{QUESTION_ID}/{N} \
       --seed 0 > /tmp/judge-{N}.json
   ```

2. Materialise the prompt to a text file for inspection / dispatch:

   ```bash
   python3 -c "import json; \
   print(json.load(open('/tmp/judge-{N}.json'))['prompt_text'], end='')" \
       > /tmp/judge-{N}.txt
   ```

3. Dispatch the judge **subagent** (the contract the pre_tool hook
   relies on — get this prefix exactly right):

   - `subagent_type`: `general-purpose`
   - `description`: `scripts_eval judge: {TARGET}/{QUESTION_ID}/{N}`
   - `prompt`: the verbatim contents of `/tmp/judge-{N}.txt`

4. Capture the subagent's final-text response and record:

   ```bash
   A_LABEL=$(jq -r .blind_label_for_A /tmp/judge-{N}.json)
   C_LABEL=$(jq -r .blind_label_for_C /tmp/judge-{N}.json)
   # Pipe the subagent verdict text on stdin:
   uv run --group experiments python -m experiments.scripts_eval.judge \
       record --run $SEER_EVAL_RUN_ID \
       --pair-key {TARGET}/{QUESTION_ID}/{N} \
       --blind-label-for-a "$A_LABEL" \
       --blind-label-for-c "$C_LABEL" \
       --verdict-file -
   ```

   If `record` exits non-zero with a `non-JSON` or vocabulary error,
   re-dispatch the judge subagent for that trial and re-record. Do
   **not** silently fall back to a tie — the script refuses to
   guess. `record` is idempotent, so the second attempt overwrites
   cleanly.

**After all 3 trials judged**, run validate + summarize + commit:

```bash
uv run --group experiments python -m experiments.scripts_eval.validate \
    --run $SEER_EVAL_RUN_ID

uv run --group experiments python -m experiments.scripts_eval.summarize \
    --run $SEER_EVAL_RUN_ID \
    --out docs/eval-rounds/2026-05-15-round-01.md

git add docs/eval-rounds/2026-05-15-round-01.md
git commit -m "round-01: completed {TARGET}/{QUESTION_ID} (both arms + judge)"
```

Report back: judge winners on the set (A/C/tie counts), whether arm C
actually used `repo-map` scripts (look at the `### tools_used` of each
arm-C cell or at the captured tools_used metric in the cell JSON), and
the next pending set per the run-state table.
